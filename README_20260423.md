# ACC102 A-Share Volatility System

## Objective
Build a traceable and reproducible A-share volatility forecasting project for ACC102, using a lightweight `B + B` route first:

- curated liquid stock universe rather than official CSI 300 constituents
- AKShare-first data collection rather than Tushare-first production route

The current objective is not to claim a finished institutional-grade system, but to deliver a working engineering pipeline with a clear upgrade path.

## Current Route And Status
Current route: `B_B_LIGHTWEIGHT`

Confirmed current state:

- `49`-stock curated liquid universe has been built from AKShare market snapshot.
- `49`-stock raw daily price fetch has completed successfully.
- downstream modeling has now been rerun on the full-universe pipeline.
- the first sparse non-price feature layer exists via individual fund flow data, but is currently dropped by coverage gating.
- a broader macro-rate non-price feature layer has been added and is currently usable in the full-universe model path.
- regularized and boosted models both run; regularized is the current default app model, while boosted now shows mixed evidence across holdout and walk-forward evaluation.
- Streamlit app is an MVP reader of persisted artifacts, not an online training service.

This means the project is now executable end to end on the current lightweight route, but model choice and feature stability still need refinement.

## Main Workflow
The project currently follows this path:

1. Build curated universe
2. Fetch adjusted daily prices (`qfq`)
3. Merge per-stock files into a panel
4. Build target `FUTURE_RV_5`
5. Build price features and merge first non-price feature layer
6. Train baseline, regularized, and boosted models
7. Compare models
8. Read persisted outputs in Streamlit app

The new walk-forward evaluator adds a lightweight temporal robustness check on top of the existing feature panel.

```powershell

```

## Current Modeling Interpretation
At the current stage, the engineering result is more important than claiming strong predictive power.

What is true right now:

- regularized ridge is the final submission model on the main held-out benchmark
- the current boosted configuration improves over the older boosted branch and is slightly better on aggregate walk-forward RMSE, but it is retained only as a comparison branch
- fund flow integration is preserved in the data layer, but not yet stable enough for the default model path
- macro-rate enrichment is currently the more usable non-price supplement
- therefore model quality should still be interpreted cautiously

This is an expected intermediate state in a lightweight route, not a failure of the project.

## Walk-Forward Evaluation
The walk-forward layer currently uses:

- the latest `feature_panel_int`
- the plain linear baseline, the regularized ridge branch, and the tuned boosted branch
- expanding train window plus rolling forward test blocks

Its purpose is to answer:

- does the model retain any stability across time splits
- how sensitive are results to limited recent coverage
- whether the current full-universe lightweight setup is robust enough to justify further feature expansion

## Known Limitations
- current route is still `B + B`, not the more stable `A + A` route
- boosted tuning is no longer missing, but its ranking relative to regularized remains mixed across evaluation lenses
- only one broad-coverage non-price feature layer is currently used in the model path
- post-submission feature expansion is still optional if we want to revisit boosted later
- GitHub / Streamlit deployment is still user-owned

## Recommended Next Steps
Recommended priority order:

1. complete GitHub and Streamlit deployment using the deployment handoff
2. keep regularized as the final submission model
3. optionally expand non-price features after deployment if you want a post-submission improvement cycle
